# Project Review: `meet`

## Executive Summary

Well-built MVP with clean architecture and minimal dependencies. The Swift+TypeScript dual-layer design via file-based IPC is sound. Main concerns: a stale bug in `status.ts`, quadratic performance in the pipeline, missing config validation, and several silent error-swallowing patterns on critical write paths.

---

## Bugs (Fix Immediately)

### 1. `status.ts:90` ā€” Reads `/tmp` instead of `~/.meet/sessions/`

`findSessions()` hardcodes `"/tmp"` but sessions are stored at `~/.meet/sessions/` (per `storage.ts:31`). The `meet status` command **cannot find any current sessions**.

**Fix**: Replace line 90 `const tmpDir = "/tmp"` with `const tmpDir = getSessionsDir()` and add the import.

### 2. `cli.ts:408` ā€” Misleading rename of async `readdir` as `readdirSync`

```typescript
const { readdir: readdirSync } = await import("node:fs/promises")
```

This is async but named `readdirSync`. Also redundant ā€” `readdir` is already imported statically on line 11.

**Fix**: Remove the dynamic import; use the existing static import.

### 3. `assembler.ts:49` ā€” `appendEntry` accepts unused `header` parameter

The function signature includes `header` but never uses it. Called from `recorder.ts:89` passing `header` unnecessarily.

**Fix**: Remove the `header` parameter from `appendEntry` and update the call site.

## Performance Issues

### 4. `pipeline.ts:114-118` ā€” O(n) `isProcessed` on every enqueue

```typescript
private isProcessed(key: string): boolean {
  return this.session.processedChunks.some(
    (c) => `${c.source}-${String(c.index).padStart(3, "0")}` === key && c.status === "done"
  );
}
```

Called for every WAV file on rescan and enqueue. With 240 chunks/hour, this is quadratic.

**Fix**: Maintain a `Set<string>` of processed keys, updated when chunks complete. O(1) lookup.

### 5. `pipeline.ts:157` ā€” Config re-read from disk every chunk

`loadConfig()` reads and parses `~/.meet/config.json` on every chunk transcription. Config doesn't change mid-session.

**Fix**: Load config once at `Pipeline` initialization and reuse.

### 6. `pipeline.ts:192` ā€” Full session JSON rewrite every chunk

`writeSession()` serializes and writes the entire `Session` object (with growing `processedChunks` array) after every chunk. 240 writes for a 1-hour session, each writing a larger file.

**Fix**: Consider debounced writes or delta updates.

## Type Safety Issues

### 7. `recorder.ts:132-136` ā€” `as any` casts on CaptureEvent

```typescript
if ((ev as any).source === "mic") { ... }
```

`CaptureEvent` in `capture-events.ts:7` uses an index signature `[key: string]: unknown` as an escape hatch, undermining the discriminated union.

**Fix**: Use proper type narrowing. Remove the index signature or add explicit property types.

### 8. `types.ts` vs `audio-metrics.ts` ā€” Duplicate `AudioMetrics` interface

Two interfaces with the same name but different shapes (`sampleCount` is `number` vs `number?`).

**Fix**: Consolidate into one in `types.ts`.

## Error Handling Issues

### 9. Silent `.catch(() => {})` on critical write paths

| Location | What's swallowed | Impact |
|---|---|---|
| `recorder.ts:89` | `appendEntry` failure | Empty transcript, zero signal |
| `pipeline.ts:192` | `writeSession` failure | Session state lost on disk error |
| `finalize.ts:39` | Progress update write | User sees stale progress |
| `finalize.ts:225,235` | Cleanup / error-state writes | Corrupted session state |

**Fix**: At minimum, log to stderr. For session state writes, consider retry or error propagation.

### 10. `finalize.ts:113`, `storage.ts:92` ā€” Unchecked `JSON.parse`

Corrupt `session.json` causes unhandled throw. No try/catch around the parse.

**Fix**: Wrap in try/catch, return null on parse failure, handle downstream.

### 11. `process.exit()` in library code

`cli.ts:123,134`, `import.ts:44,50,58,64`, `recorder.ts:162` ā€” these prevent callers from handling errors gracefully and make code untestable.

### 12. `process.exit(0)` can abort in-flight async writes

`recorder.ts:256,261`, `cli.ts:257` ā€” calling `process.exit()` during async shutdown can leave `session.json` partially written.

## Code Quality Issues

### 13. `storage.ts:36-76` ā€” 40-line repetitive `loadConfig` merge chain

Each config property is manually merged with `??` chains. Fragile ā€” adding a field requires updating the chain.

**Fix**: Use `Object.assign({}, DEFAULT_CONFIG, fileConfig, overrides)` or spread operator.

### 14. `phrasebook.ts:98-100` ā€” Duplicate `expandPath` function

Identical copy of `storage.ts:26-28`. DRY violation.

**Fix**: Import from `storage.ts`.

### 15. `phrasebook.ts:102-103` ā€” Module-level singleton cache

`_cached` and `_cachedPath` are global mutable state, making testing harder.

### 16. `finalize.ts:174,187,194,202` ā€” Copy-paste `mergedResults` pattern

The same `Map` merge pattern appears 4 times. Extract to helper.

### 17. `pipeline.ts:219-229` ā€” Busy-wait `drainQueue` polling

`setTimeout(100)` spin-lock. `processNext()` already self-chains via recursive call. Two drivers for one queue.

## What's Done Well

- Minimal dependency footprint (chalk, chokidar, commander, nanoid)
- File-based IPC between Swift and TypeScript is crash-resilient
- Sequential whisper queue prevents GPU/CPU contention
- Atomic writes throughout (`writeAtomic` pattern)
- Lock-based concurrency control (O_EXCL exclusive create)
- Two-phase finalization with live transcript backup
- Multi-layer silence gating
- Health monitoring with auto-restart
- Strict TypeScript config
- Tests use synthetic WAV fixtures for deterministic testing
- Clean ESM module setup
